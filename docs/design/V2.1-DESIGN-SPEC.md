# V2.1 Design Specification: Flexible Shot Revision

## Status and outcome

This is a design-phase proposal, not an implementation or final schema.
V2.1 enables a single shot to be rendered as a new candidate, reviewed,
selected for a cut, and reassembled without rendering unrelated shots.

## Design rules

- The shot is the production unit; a film is a dependency graph.
- Prompts are derived execution artifacts, never the canonical source of truth.
- Metadata is file-based, structured, versioned, and uses stable IDs and ISO
  8601 UTC timestamps. Canonical manifest paths are project-root-relative.
- A render job executes work; a candidate is its immutable result. Binary
  files are not the source of status.
- A completed candidate is immutable. Reviews and editorial selections are
  separate append-only records.
- Approval does not select a candidate for a cut. A Cut Manifest does.
- Model-specific fields belong in a Render Plan `model_specific` object or an
  adapter namespace.
- Published artifacts are immutable. Editorial changes create a new cut
  version; they do not change source candidates.

## Entity contracts

These are minimum logical fields. Implementations may add fields while keeping
unknown fields forward-compatible.

### 1. Project Manifest

`drama-factory.project.json` identifies a project and its stable layout:

```text
schema_version, project_id, project_name, engine_version, project_root,
paths, default_renderer, default_audio_policy, created_at, updated_at
```

`project_root` is `.` or another relative logical root, never an absolute host
path. `paths` maps named areas such as `shots`, `cuts`, `audio`, and `delivery`
to relative paths. The manifest is human-authored project configuration;
timestamps and derived indexes may be updated by tooling.

### 2. Shot Contract

The Shot Contract is the canonical, revisioned specification of intent:

```text
schema_version, shot_id, scene_id, title, dramatic_intention, duration_target,
characters, environment, start_state, end_state, primary_action, camera,
lighting, dialogue_dependency, audio_dependency, continuity_dependencies,
references, known_risks, acceptance_criteria, status, revision
```

Human-authored fields are dramatic intention, character/environment/state,
action, camera, lighting, dependencies, risks, and acceptance criteria.
Derived fields may include reference checksums, validation results, and a
compiled prompt artifact reference. A contract revision never mutates the
historic revision used by an existing plan; it creates a new revision.

### 3. Render Plan

A Render Plan is an immutable execution snapshot for one render attempt:

```text
schema_version, render_plan_id, shot_id, renderer, model, model_version, task,
compiler_version, prompt_artifact, negative_prompt_artifact, references,
audio_conditioning, duration, resolution, fps, seed_policy, candidate_budget,
handles, model_specific, estimated_cost, created_at
```

The plan records the Shot Contract revision and checksums of referenced
artifacts in `references` or provenance. Prompt artifacts are compiled from
the contract and are auditable, not authoritative. Any change produces a new
plan ID or explicit revision; no plan file is edited in place.

### 4. Render Job

A Render Job records an execution attempt:

```text
render_job_id, render_plan_id, candidate_id, status, worker, provider,
started_at, completed_at, exit_code, runtime_seconds, estimated_cost,
actual_cost, log_path, error, retry_of
```

Lifecycle: `QUEUED` → `RUNNING` → `SUCCEEDED` or `FAILED`; `QUEUED` and
`RUNNING` may transition to `CANCELLED` with an audit reason. A candidate ID is
reserved when the job is queued, but a Candidate record is promoted only after
the complete artifact set is validated, checksummed, and atomically published.
Failed or cancelled jobs never create a reviewable Candidate.

### 5. Candidate

A Candidate is an immutable render result:

```text
schema_version, candidate_id, shot_id, render_plan_id, render_job_id, version,
status, video_path, proxy_path, thumbnail_path, metadata_path, checksum,
duration, fps, resolution, created_at, provenance, qc_summary
```

Lifecycle: `REVIEW_REQUIRED` → `PICTURE_APPROVED`, `REJECTED`, or
`SUPERSEDED`. `SUPERSEDED` is a historical classification, not deletion. A
candidate never has `SELECTED_FOR_CUT`; selection is an independent entity.
The binary and its candidate metadata cannot change after promotion. QC
summaries are referenced snapshots; new findings belong in the Review Ledger.

### 6. Review Ledger

Each review is a new append-only record:

```text
review_id, candidate_id, reviewer, reviewer_type, created_at, decision,
summary, findings
```

Each finding contains:

```text
finding_id, category, severity, start_timecode, end_timecode, frame, evidence,
confidence, suggested_action, waived, waiver_reason
```

Decisions are `APPROVE_PICTURE`, `REQUEST_REVISION`, `REJECT`, and
`COMMENT_ONLY`. The candidate status is a derived current assessment from the
authoritative review policy, but the ledger never overwrites prior reviews.
Conflicting reviews remain visible; a named authorized human resolution is a
new review/decision record that references the earlier review IDs.

### 7. Candidate Selection

Selection is an append-only editorial decision:

```text
selection_id, shot_id, candidate_id, selected_by, selected_at, purpose, notes,
supersedes_selection_id
```

`purpose` is one of `ROUGH_CUT`, `FINAL_CUT`, `BENCHMARK`, or `PREVIEW`.
A replacement selection points to the selection it supersedes. Selection is
per purpose and has no effect on candidate approval status.

### 8. Cut Manifest

The Cut Manifest is the editorial source of truth:

```text
schema_version, cut_id, project_id, cut_name, cut_version, status, timeline,
audio, created_at, created_by, parent_cut_id
```

Each timeline item contains:

```text
sequence_index, shot_id, candidate_id, source_in, source_out, timeline_in,
timeline_out, transition, audio_package_id, notes
```

Cut status is `DRAFT`, `ROUGH_CUT`, `REVIEW`, `PICTURE_LOCKED`, or
`DELIVERY_APPROVED`. Picture lock belongs to the Cut Manifest, never a
candidate. Any editorial modification creates a new cut version with
`parent_cut_id`, leaving previous cut manifests immutable.

### 9. Shot Package

A Shot Package is an immutable official handoff to editorial or audio:

```text
shot_package_id, shot_id, candidate_id, picture_status, audio_status,
video_source, proxy, handles, dialogue_stem, foley_stem, local_sfx_stem,
provenance, published_at
```

It is created only after its candidate and intended source components pass
validation. A revised package receives a new ID; it never replaces an existing
package. It may exist before a candidate is selected for a cut, but a Cut
Manifest must name the exact package/audio version it consumes.

## States and permitted transitions

```text
Render Job:  QUEUED -> RUNNING -> SUCCEEDED
                         |          
                         +-> FAILED / CANCELLED

Candidate:   REVIEW_REQUIRED -> PICTURE_APPROVED
                    |              
                    +-> REJECTED / SUPERSEDED

Publish:     selection -> SHOT_PACKAGE_APPROVED -> Cut PICTURE_LOCKED
                                             -> DELIVERY_APPROVED
```

The publish line is a relationship between records, not a status placed on a
candidate. Validation rejects a selected rejected candidate and rejects a
picture-locked cut with unresolved references.

## Project layout

```text
project-root/
├── drama-factory.project.json          # project manifest
├── project/
│   ├── scenes/                         # human-authored scene source metadata
│   ├── shots/shot-011/
│   │   ├── contract/                   # revisioned canonical contracts
│   │   ├── render-plans/               # immutable plan snapshots
│   │   ├── jobs/                       # job state and logs metadata
│   │   ├── candidates/cand-shot011-002/# immutable result plus sidecar metadata
│   │   ├── reviews/                    # append-only ledger entries
│   │   ├── selections/                 # append-only editorial choices
│   │   └── publishes/                  # immutable Shot Packages
│   ├── cuts/                           # immutable Cut Manifest versions
│   ├── audio/                          # versioned production audio and packages
│   └── delivery/                       # immutable delivery outputs and manifests
└── .drama-factory/
    ├── cache/                          # disposable, ignored cache
    ├── runtime/                        # temporary execution state, ignored
    └── locks/                          # short-lived locks, ignored
```

The proposed layout is retained with two constraints: metadata lives next to
the entity it describes, and runtime state never becomes canonical source.
Candidate directories contain generated source video and optional review proxy
or thumbnail, while candidate metadata records their checksums. Review proxies
are derivative artifacts; they can be regenerated. Published Shot Packages and
delivery outputs are immutable. Do not create folders for future departments
until an entity needs them.

## Validation rules

- Every JSON/YAML metadata file declares a supported `schema_version`, stable
  ID, UTC timestamps where applicable, and relative paths.
- IDs are unique within their entity namespace; IDs embedded in a path must
  equal the ID inside metadata.
- A plan points to an existing Shot Contract revision and checksummed inputs.
- A successful job has complete times, valid artifacts, and a matching plan and
  reserved candidate ID before candidate promotion.
- Candidate media must exist, decode, match declared duration/fps/resolution,
  and match checksum. A proxy is optional but must validate if declared.
- A review may only reference an extant, checksum-valid candidate.
- Selection requires an extant candidate with `PICTURE_APPROVED` for a normal
  rough/final cut; benchmark and preview policy may be explicitly looser.
- A Cut Manifest has unique sequence indexes, valid ranges, monotonic timeline
  ranges, no missing candidates/packages, and no rejected selected candidate.
- Assembly validates the Cut Manifest and every checksum before it starts.

## Failure, recovery, and audit behaviour

| Scenario | File-based response |
| --- | --- |
| Render fails or is interrupted | Mark job `FAILED` or `CANCELLED`; preserve log and error; leave temporary output outside candidate directory; retry with a new job linked by `retry_of`. |
| Video is missing, corrupt, or metadata arrives first | Do not promote Candidate. Validate a temporary artifact, write sidecar, checksum, then atomically rename/publish the complete directory. |
| Candidate ID collision | Acquire a per-shot lock, reject the collision, record the failed reservation, and allocate a new stable ID. Never reuse an ambiguous ID. |
| Review targets a missing candidate | Reject the review write; retain an audit event and require restoration from immutable storage before review. |
| Approved candidate checksum changes | Treat as integrity failure; block selection/assembly, mark audit finding, and restore or create a new candidate—never rewrite the original record. |
| Selected candidate is manually deleted | Cut validation fails with a missing artifact error; restore exact checksum or create a new cut selecting another approved candidate. |
| Cut references rejected candidate | Validation fails; create a replacement selection/cut version. |
| Contract changes after rendering | Preserve old contract and plan provenance; create new plan/candidate only if the change warrants rerendering. |
| Model version unknown | Block production rendering; a legacy import may record `UNKNOWN` with an explicit provenance limitation. |
| Assembly fails mid-run | Write outputs to a temporary assembly directory, retain logs, then atomically publish delivery only after all shots and final validation succeed. |

Atomic write means writing metadata or artifacts to a same-filesystem temporary
path, fsyncing where available, validating, then renaming into final location.
Locks are short-lived coordination files with owner and expiry metadata. Audit
records are append-only metadata events, not a database requirement.

## Command experience (design only)

These names describe a future interface; V2.1 does not implement a CLI.

| Command | Input and validation | Creates / changes | Idempotence and failure |
| --- | --- | --- | --- |
| `drama inspect` | Project root; reads manifest and layout | No state | Idempotent; fails on missing/invalid manifest. |
| `drama validate` | Project, shot, or cut scope; applies validation rules | Optional report only | Idempotent; returns diagnostics without mutation. |
| `drama shot show shot-011` | Stable shot ID | No state | Idempotent; fails if contract is absent. |
| `drama render shot-011` | Existing approved plan or explicit new-plan intent | Reserves job/candidate; job state advances | Retry-safe by explicit job ID; fails on invalid plan, lock, or unknown model. |
| `drama render shot-011 --new-plan` | Contract revision plus explicit changes | New immutable Render Plan and job reservation | Never edits a plan; fails on missing required approved inputs. |
| `drama candidates shot-011` | Shot ID | No state | Idempotent; lists immutable candidate metadata. |
| `drama review shot-011 cand-shot011-002` | Candidate and reviewer decision/findings | New ledger entry; derived status may change | Repeated command creates distinct reviews unless an idempotency key is supplied; fails on integrity error. |
| `drama approve shot-011 cand-shot011-002` | Authorized human and approval record | New `APPROVE_PICTURE` review record | Same as review; fails on missing QC or checksum mismatch. |
| `drama select shot-011 cand-shot011-002 --cut rough-cut-v2` | Approved candidate and named purpose/cut | New Selection and a new Cut Manifest version | Idempotent only with selection key; fails on rejected/missing candidate. |
| `drama cut validate rough-cut-v2` | Cut ID/version | No state | Idempotent; fails with precise broken reference diagnostics. |
| `drama assemble rough-cut-v2` | Validated Cut Manifest and artifacts | Temporary assembly then immutable delivery record | Repeated identical inputs may return existing checksum-matching delivery; otherwise fails atomically. |

The command set is intentionally grouped by user intent. `approve` is retained
only as a convenience form of `review`; it must still write a ledger record.
`select` should always name a cut or purpose to avoid implying that approval is
editorial selection.

## V2.1 scope boundary

In scope: file-based project metadata, contracts, plans, single-shot
orchestration, immutable candidates, review ledger, selection, Cut Manifest,
assembly from selected candidates, minimum provenance/validation, and Shot 11
migration.

Out of scope: database, web UI, authentication, distributed queue, multi-GPU
scheduling, autonomous director, automatic repair, full continuity graph,
character memory, multi-model routing, automatic billing, and cloud control
plane.
