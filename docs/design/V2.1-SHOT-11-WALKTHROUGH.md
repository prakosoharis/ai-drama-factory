# Historical Use Case: V1 Shot 11 Flexible Revision

This walkthrough preserves the V1 Shot 11 scenario as a historical migration
use case. It is not the canonical V2 example or a required implementation
dependency; the native V2 workflow uses a new clean creative project. Nothing
is rendered or written to `project-film`.

| Step | Entity and state | Files created or changed | Result |
| --- | --- | --- | --- |
| 1. Establish intent | `shot-011`, Contract revision `r001`, `APPROVED_FOR_RENDER` (policy-defined contract status) | `project/shots/shot-011/contract/shot-011-r001.json` | Human-authored intent, references, acceptance criteria, and constraints are canonical. |
| 2. Plan first attempt | `rp-shot011-001`, immutable snapshot | `render-plans/rp-shot011-001.json`; derived prompt artifact referenced by checksum | Plan compiles Contract `r001` for the chosen renderer. |
| 3. Start execution | `job-shot011-001`: `QUEUED` → `RUNNING`; candidate ID reserved: `cand-shot011-001` | `jobs/job-shot011-001.json`; short-lived lock in `.drama-factory/locks/` | Worker/provider details and start time are recorded separately from candidate data. |
| 4. Promote result | Job `SUCCEEDED`; candidate `cand-shot011-001`: `REVIEW_REQUIRED` | `candidates/cand-shot011-001/video.mp4`, sidecar metadata, optional proxy/thumbnail, checksum | Temporary output is decoded and checksummed before atomic promotion. |
| 5. Reject first result | Review `rev-shot011-001`; candidate derived status `REJECTED` | `reviews/rev-shot011-001.json` | Human or Review Critic records excessive head movement with a timecoded finding. Candidate binary and metadata remain unchanged. |
| 6. Preserve evidence | Finding `find-shot011-001` has timecodes, evidence, confidence, and suggested action | No candidate change; optional linked evidence derivative | The ledger remains append-only and tells later reviewers why revision is needed. |
| 7. Create revision plan | `rp-shot011-002`, new plan based on Contract `r001` plus explicit motion constraint | `render-plans/rp-shot011-002.json`; new compiled prompt artifact | The original plan is not edited. The new plan records what changed and uses a new ID. |
| 8. Render second candidate | `job-shot011-002`: `QUEUED` → `RUNNING` → `SUCCEEDED`; `cand-shot011-002`: `REVIEW_REQUIRED` | New job record and `candidates/cand-shot011-002/` artifact set | No other shot is queued or rerendered. |
| 9. Approve picture | Review `rev-shot011-002`: `APPROVE_PICTURE`; candidate derived status `PICTURE_APPROVED` | `reviews/rev-shot011-002.json` | Approval is a ledger decision; it does not select the candidate for the film. |
| 10. Choose for rough cut | `sel-shot011-002-rough`, purpose `ROUGH_CUT` | `selections/sel-shot011-002-rough.json` | An explicit editorial decision points from shot 011 to candidate 002. |
| 11. Revise cut | `rough-cut-v2`, version `002`, status `ROUGH_CUT`, parent version `001` | `project/cuts/rough-cut-v2/v002.json` | Only the Shot 11 timeline item changes from its prior candidate ID to `cand-shot011-002`. |
| 12. Assemble | Assembly input validates; temporary assembly succeeds | Temporary assembly directory, immutable delivery manifest/output after validation | The assembler reads `v002` and selected artifacts; it renders no source shot. |
| 13. Retain history | Candidate 001 remains `REJECTED`; candidate 002 remains `PICTURE_APPROVED` | No historic file changes | Both results, plans, jobs, and reviews remain auditable. |

## Required state at the key decision points

```text
job-shot011-001       SUCCEEDED
cand-shot011-001      REJECTED
rev-shot011-001       REQUEST_REVISION

job-shot011-002       SUCCEEDED
cand-shot011-002      PICTURE_APPROVED
rev-shot011-002       APPROVE_PICTURE
sel-shot011-002-rough purpose=ROUGH_CUT, candidate=cand-shot011-002
rough-cut-v2/v002     ROUGH_CUT, timeline[shot-011]=cand-shot011-002
```

## Review finding example

```text
finding_id: find-shot011-001
category: motion
severity: major
start_timecode: 00:00:01:12
end_timecode: 00:00:02:08
frame: 36
evidence: review-proxy frame reference
confidence: high
suggested_action: constrain head rotation and reduce action amplitude
waived: false
```

## Why the revision is selective

The Cut Manifest identifies each source candidate explicitly. Reassembly reads
the chosen candidates and audio packages; it does not discover files by naming
convention or invoke any renderer. As a result, the only costly work is the
second render job for Shot 11. A new delivery is still required, because the
finished sequence has changed, but it is assembly work rather than shot
generation.
